## 결정/근거 요약 (AI 검색용 키워드 포함)

### 1. v1.10.0 의 release integrity 버그 발견 경로

외부 정적 리뷰가 plan 문서와 실제 코드 mismatch 를 짚었다. 핵심:

- **CHANGELOG / SKILL.md** 는 "Schema v5, learnings_meta 도입" 으로 표기
- **실 코드**: `harness-state.sh:12 HS_SCHEMA_VERSION=4`, `:16 STAMPED="3 4"`, `hs_migrate_v4_to_v5()` 정의는 있으나 **호출처 grep 결과 0건 (dead code)**
- 결과: `hs_write_manifest` 가 새 매니페스트에 `schema_version=4` 를 stamping → v5 라고 광고했지만 실제로는 v4

**근거 자료**: 평가가 `harness-state.sh:12,16` 와 `SKILL.md:2112` 의 "schema_version: 4 + ehr_cycle 필드" 표현을 동시 인용. 사실 검증 (`grep -nE "HS_SCHEMA_VERSION|hs_migrate_v4_to_v5"`) 으로 즉시 확인 가능.

### 2. 사용자 정책: 최소 수정 + 사내 lightweight

평가는 P0~P2 로 다양한 권고 (CI, governance 7단계 lifecycle, confidence 필드, registry 외부화, `/ehr:doctor`, golden snapshot 등) 를 제시. 사용자는 "최소 수정" + "큰 개편은 테스트가 힘들어서 원하지 않음" 명시.

→ **채택**: P0 정합성 버그 (해당 라인만 수정)
→ **기각**: CI / 7단계 lifecycle / Java AST / registry 외부화 / `/ehr:doctor` / golden snapshot / onboarding 문서 — 사내 lightweight 도구 scope 외, 큰 인프라

이 판단 기준은 향후 평가 받을 때 동일하게 적용. **사용자 의도(개인용 사내 도구) ≠ 평가자 의도(사내 표준화)** 차이를 항상 먼저 분리.

### 4. v1.10.2 의 잔여 P0 (Step 4-G validation)

평가가 1 라운드 더 짚은 진짜 잔여 버그:

- `SKILL.md:2194`: `if (m.schema_version !== 4) throw ...` — `hs_write_manifest` 가 v5 stamping 한 직후 검증이 v4 expected → 정상 매니페스트가 "유효성 검증 실패" 경고
- 동일 블록 echo `schema_version=4` 도 stale
- audit 설명 `Step 0.7-C` 의 `{3, 4}` 도 잔존

이건 manifest 파일을 망가뜨리지는 않지만 **신뢰도/UX** 문제. 평가의 표현: "P0- / P1+". 한 commit 으로 처리.

### 5. v1.10.2 의 README 깔끔 정리 + landing 미러

- 히스토리(변경 이력 섹션)는 보존 정책으로 유지
- 정합성 5 곳만 수정: 현재 릴리즈 / 목차 / 디렉토리 구조도 / §12 제목·본문 / 마이그레이션 절
- v5 신규 단락(`learnings_meta` JSON + 정책 설명) 한 단락 추가
- landing 의 `public/README_HARNESS.md` 는 1:1 복사
- landing 의 `public/README_HARNESS.html` 은 marked + GitHub-style heading slug awk 로 article 영역 재변환, wrapper(topbar/meta/style/script) 보존
- HTML heading id 부여 awk 패턴: `tolower()` + `[.,?!()\[\]:;]` 제거 + 공백 → 하이픈 → 한글은 그대로 유지

### 6. 마켓플레이스 카탈로그 정합성 (마지막 라운드)

평가 마지막 지적:

- `.claude-plugin/marketplace.json` 의 `entry version: "1.9.4"` 가 `plugin.json (1.10.2)` 와 어긋남
- `metadata.pluginRoot: "./plugins"` + `source: "./plugins/ehr-harness"` 동시 사용은 spec 상 모호 (`./plugins/./plugins/ehr-harness` 가능)
- Claude Code 공식: plugin.json::version 1순위, marketplace entry 보조 → 런타임 차단은 아니지만 카탈로그 정합성 문제

**채택**: 평가의 "첫 번째 권장안" — entry version 제거 + pluginRoot 제거 + source 표준 relative path 유지. 결과는 공식 문서의 "같은 저장소 단일 플러그인" 예시와 일치.

**plugin.json bump 생략** — 카탈로그 메타만 변경, 런타임 동작 변경 없음. CHANGELOG `[Unreleased]` 에 한 줄 기록.

## 패턴 메모 (재사용 가능한 작업 흐름)

### 평가 비판적 필터링 단계

1. 평가 주장을 grep / 코드 검증으로 **사실 여부** 먼저 확인 (평가가 틀렸을 수도 있음)
2. **사용자 의도** 기준으로 권고를 채택/기각 분류 (사내 lightweight ≠ 표준화)
3. 채택분만 **한 atomic commit** 으로 묶고 회귀 PASS 확인 후 push
4. 의도적 deviation 은 CHANGELOG 또는 보류 메모에 명시

### 1 day 멀티-라운드 push 흐름

worktree 격리 → 새 브랜치 → 작업 → 회귀 → atomic commit → branch push → main fast-forward → main push → worktree+branch 정리. 평가 라운드마다 동일 패턴 반복.

### Markdown ↔ HTML 미러 동기화

- `.md` 는 단순 복사
- `.html` 은 marked CLI (`-i input.md --gfm`) → article body 추출 → awk 로 heading id 부여 → 기존 HTML wrapper 의 `<article>` 사이만 교체
- heading slug 규칙: `tolower()` 만 적용, 한국어는 그대로, punctuation 제거, 공백 → 하이픈, 양끝 하이픈 trim
